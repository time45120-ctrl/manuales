Sección 1

Manual de despliegue

1. Introducción

El despliegue del proyecto “La IA como cerebro, orquestador y controlador de los dispositivos IoT” representa el paso desde un entorno de desarrollo local hacia una infraestructura de producción temprana.

En esta etapa, el sistema ya cuenta con componentes publicados y comunicados entre sí: una aplicación web pública, una API backend accesible por Internet, una base de datos en la nube, servicios de inteligencia artificial y un primer dispositivo IoT físico basado en ESP32.

El objetivo del despliegue no es presentar el sistema como una plataforma final totalmente madura, sino como una solución funcional en proceso de consolidación. Actualmente, el proyecto permite validar el ciclo principal de operación: captura de voz, transcripción, interpretación mediante IA, generación de un plan de acción, confirmación por parte del usuario, encolamiento del comando, consulta desde el ESP32, ejecución física y confirmación de resultado.

Esta versión funciona como una SPA básica: el documento no recarga la página, solo cambia la sección visible al hacer clic en el sidebar.
Sección 2

2. Visión general del despliegue

La solución desplegada se organiza como una arquitectura distribuida. Cada componente se aloja o ejecuta en un entorno especializado, de acuerdo con su responsabilidad dentro del sistema.

La topología general del despliegue es la siguiente:

Usuario
  -> Frontend público en Hostinger
     https://afcrseguridad.com
  -> Backend público en AWS
     https://api.afcrseguridad.com
  -> Servicios de inteligencia artificial OpenAI
  -> Base de datos y almacenamiento Supabase
     Proyecto: omkbowrspgbuwpifksfk
  -> Dispositivo IoT ESP32
     Polling HTTP(S) + ACK

El frontend funciona como capa de interacción con el usuario. El backend actúa como núcleo lógico y orquestador. Supabase conserva la información persistente y aplica reglas de seguridad. OpenAI proporciona capacidades de transcripción e interpretación de lenguaje natural. El ESP32 funciona como primer dispositivo físico integrado para validar la ejecución real de comandos.

El flujo desplegado puede describirse de la siguiente manera:

  1. El usuario ingresa al sitio público mediante HTTPS.
  2. La aplicación web verifica la disponibilidad del backend.
  3. El usuario graba un comando de voz desde el navegador.
  4. El frontend envía el audio al backend mediante la API pública HTTPS.
  5. El backend transcribe e interpreta el comando con apoyo de OpenAI.
  6. El backend genera una respuesta humana y un JSON técnico.
  7. El sistema crea un plan pendiente de confirmación.
  8. El usuario confirma la ejecución desde el dashboard.
  9. El backend registra y encola el comando para el dispositivo.
  10. El ESP32 consulta la API para recibir comandos pendientes.
  11. El ESP32 ejecuta la acción física.
  12. El ESP32 envía un ACK de confirmación al backend.
  13. El dashboard muestra el estado final del comando.
Este flujo demuestra que la inteligencia artificial no ejecuta directamente una acción física sin control. La IA interpreta y propone; el backend valida y organiza; el usuario confirma; el dispositivo ejecuta; y la plataforma registra el resultado.
Sección 3

3. Despliegue del frontend

El frontend público del proyecto se encuentra desplegado en Hostinger bajo el dominio: 

https://afcrseguridad.com

Esta aplicación fue desarrollada con Next.js, React, TypeScript y Tailwind CSS. Su función principal es ofrecer una interfaz web para que el usuario pueda acceder al sistema, grabar comandos de voz, revisar la respuesta generada por la IA, confirmar acciones y sincronizar dispositivos IoT.

En el despliegue actual, Hostinger está configurado como entorno para una aplicación Next.js. La configuración correcta corresponde a un servidor Next.js ejecutado sobre Node.js, no a una exportación estática pura.

El proceso de construcción del frontend se realiza con:

cd /home/abraham/proy_ia_security/frontend
npm run build

El proceso de arranque se realiza con:

npm run start

El comando de inicio ejecuta server.js, archivo encargado de iniciar Next.js sobre la carpeta .next. El servidor escucha en el puerto indicado por la variable PORT o, si no existe, en el puerto 3000.

La variable de entorno principal del frontend es:

NEXT_PUBLIC_API_BASE_URL=https://api.afcrseguridad.com
En producción, esta URL debe mantenerse bajo HTTPS. Si el frontend público se carga por HTTPS pero intenta consumir una API HTTP, el navegador puede bloquear la solicitud por contenido mixto.

En los logs correctos del despliegue del frontend deben observarse marcas como:

AFCR_FRONTEND_BUILD=f.37
AFCR_FRONTEND_MODE=next-server

El backend concentra la lógica principal del sistema. Recibe solicitudes desde el frontend, procesa audios, se comunica con OpenAI, crea planes pendientes de confirmación, registra información en Supabase, administra dispositivos y expone los endpoints que el ESP32 consulta para recibir comandos.

Sección 5

5. Despliegue de Supabase

Supabase es la plataforma principal de persistencia del proyecto en producción temprana. El proyecto autorizado para esta aplicación corresponde a la referencia: 

omkbowrspgbuwpifksfk

Supabase proporciona base de datos PostgreSQL, autenticación, políticas de seguridad a nivel de fila, almacenamiento privado y capacidades que permiten organizar la información de la plataforma de forma segura.

En el despliegue actual, Supabase almacena:

  • Empresas y usuarios.
  • Dispositivos registrados o enlazados.
  • Tokens temporales de emparejamiento.
  • Planes generados por voz.
  • Comandos enviados a dispositivos.
  • Estados de ejecución.
  • Auditoría de voz.
  • Audios en almacenamiento privado.

El almacenamiento privado de audio se realiza en el bucket:

voice-audio
Este almacenamiento permite conservar evidencia del flujo de voz, aunque debe gestionarse con cuidado porque los audios pueden contener información sensible.
Sección 6

6. Despliegue de los servicios de inteligencia artificial

El backend consume servicios de OpenAI para realizar dos tareas principales: transcripción de audio e interpretación de intención.

La transcripción convierte el audio grabado por el usuario en texto. Luego, la interpretación transforma ese texto en una respuesta natural y en un JSON técnico orientado a automatización.

El modelo principal de transcripción configurado es:

gpt-4o-mini-transcribe

El modelo de respaldo remoto configurado es:

whisper-1

La integración con OpenAI ya fue validada en producción temprana mediante un audio de prueba con la frase prende el LED.

  • Respuesta IA para el usuario: texto natural, claro y comprensible.
  • Respuesta JSON para el dispositivo: estructura técnica usada por el backend.
Sección 7

7. Despliegue para ESP32

El ESP32 es el primer dispositivo IoT físico integrado en el despliegue de producción temprana. Su función actual es validar la ejecución de comandos físicos mediante el control de un LED, utilizando el flujo completo de emparejamiento, polling y confirmación.

En producción, el ESP32 debe comunicarse con:

https://api.afcrseguridad.com

La comunicación se realiza mediante polling HTTP(S). Esto significa que el dispositivo no espera una conexión entrante desde el backend, sino que consulta periódicamente la API para saber si tiene comandos pendientes.

El endpoint de consulta es:

GET /device/commands?device_id={device_id}
Authorization: Bearer <device_api_key>

Cuando el ESP32 recibe un comando, ejecuta la acción física y envía un ACK:

POST /device/commands/{command_id}/ack
Durante pruebas locales de laboratorio, el ESP32 no debe usar localhost. Debe usar una URL LAN accesible desde la red WiFi donde se encuentra el dispositivo.

Ejemplo de URL local correcta:

http://192.168.0.5:8000

Antes de cargar el firmware, se recomienda validar desde un celular:

http://192.168.0.5:8000/ping
Sección 8

8. Validación post-despliegue

Después de desplegar o actualizar componentes, se deben realizar pruebas por capas para identificar rápidamente dónde podría encontrarse un problema.

La validación debe realizarse por capas: frontend, backend, flujo de voz y ESP32.
Sección 8.1

8.1 Validación del frontend

Para verificar que el sitio público responde:

curl -I https://afcrseguridad.com

También deben revisarse los logs del hosting para confirmar:

AFCR_FRONTEND_BUILD=f.37
AFCR_FRONTEND_MODE=next-server

Estas marcas indican que la versión correcta fue construida y que el modo de ejecución corresponde al esperado.

Sección 8.2

8.2 Validación del backend

Para verificar la disponibilidad de la API pública:

curl https://api.afcrseguridad.com/ping

Si el endpoint responde correctamente, el backend está activo. Si no responde, se debe revisar el servicio en AWS, el dominio, el certificado HTTPS, el puerto de escucha o las variables de entorno.

Sección 8.3

8.3 Validación del flujo de voz

La validación funcional del flujo de voz requiere:

  1. Ingresar al dashboard web.
  2. Permitir el uso del micrófono.
  3. Grabar un comando corto, por ejemplo prende el LED.
  4. Verificar que el frontend envíe el audio al backend.
  5. Confirmar que la IA devuelva una respuesta natural.
  6. Confirmar que el sistema muestre un JSON técnico separado.
  7. Revisar si el plan es ejecutable.
  8. Confirmar la ejecución desde la interfaz.
Sección 8.4

8.4 Validación con ESP32

La validación física requiere:

  1. Generar un token temporal de emparejamiento.
  2. Insertar el token en el sketch del ESP32.
  3. Configurar SSID, password WiFi y URL de API.
  4. Cargar el sketch por USB desde Arduino IDE.
  5. Confirmar que el ESP32 reclame el dispositivo.
  6. Enviar un comando de voz ejecutable.
  7. Confirmar el plan desde el dashboard.
  8. Verificar que el ESP32 reciba el comando.
  9. Observar la acción física en el LED.
  10. Confirmar que el backend reciba el ACK.
  11. Revisar el estado final desde el dashboard.
Esta prueba valida el ciclo completo desde la voz del usuario hasta la acción física del dispositivo.
Sección 9

9. Riesgos y controles del despliegue

El despliegue del proyecto incorpora riesgos propios de una solución distribuida que conecta navegador, API, base de datos, servicios de IA y hardware físico. Por ello, se aplican controles técnicos y operativos.

9.1 Contenido mixto

El frontend público se sirve por HTTPS. Por esta razón, la API pública también debe utilizar HTTPS. 

NEXT_PUBLIC_API_BASE_URL=https://api.afcrseguridad.com

9.2 Variables de entorno incorrectas

Una variable mal configurada puede hacer que el frontend apunte a una API local, privada o inexistente.

Control aplicado: mantener variables separadas por entorno y validar el dominio público antes de publicar cambios.

9.3 Secretos y credenciales

Las claves de OpenAI, Supabase y otros servicios no deben almacenarse en código fuente ni exponerse en el frontend.

Control aplicado: mantener secretos en archivos de entorno o configuraciones seguras del servidor.

9.4 Conectividad del ESP32

El ESP32 puede fallar si la URL configurada no es accesible desde su red. Esto ocurre especialmente en laboratorio local si se usa localhost.

Control aplicado: en producción usar https://api.afcrseguridad.com; en local usar una IP LAN real y verificar /ping desde otro dispositivo de la red.

9.5 Ejecución no deseada de comandos

Una mala transcripción podría interpretar incorrectamente una orden.

Control aplicado: el sistema usa vista previa y confirmación. La IA no ejecuta directamente acciones físicas.

9.6 Comandos pendientes o expirados

Un dispositivo podría estar desconectado y no recibir el comando dentro del tiempo esperado.

Control aplicado: los comandos tienen un tiempo de vida limitado y pueden pasar a estado expired.

Sección 11

11. Conclusión del despliegue

El despliegue del proyecto demuestra que la solución puede operar como una plataforma distribuida en producción temprana. La aplicación web, el backend, la base de datos, la inteligencia artificial y el ESP32 trabajan de forma coordinada para transformar un comando de voz en una acción física controlada.

La separación entre frontend, backend, Supabase, OpenAI y dispositivos IoT permite que el sistema sea escalable y mantenible. Además, el uso de HTTPS, variables de entorno, confirmación previa a la ejecución, almacenamiento seguro y seguimiento de estados permite reducir riesgos durante la operación.

Esta infraestructura constituye una base sólida para continuar la evolución del proyecto hacia dispositivos más complejos, manteniendo el principio central de la tesis: la inteligencia artificial como cerebro, orquestador y controlador de un ecosistema IoT.
